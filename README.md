# 📊 EDA: Анализ данных службы поддержки для автоматизации процессов

##  Цель проекта
Провести exploratory data analysis (EDA) данных IT-поддержки для выявления проблем в бизнес-процессах обработки тикетов и формулирования системных требований к автоматизации. Проект демонстрирует подход системного аналитика к поиску «узких мест» и проектированию улучшений на основе данных.

## 📈 Ключевые инсайты

### Проблема 1: Огромный backlog незакрытых тикетов
- **67% тикетов не закрыты** (5700 из 8469)
- 34% (2881) — ожидают ответа клиента (Pending Customer Response)
- 33% (2819) — открыты без прогресса (Open)
- Только 33% (2769) — закрыты

### Проблема 2: Отсутствие логирования создания тикетов
- В системе **НЕ фиксируется дата создания тикета** (`Created At`)
- Невозможно рассчитать SLA (время от создания до решения)
- Критический пробел в системе логирования, мешающий оценке производительности

### Проблема 3: Длительное время решения
- Среднее время от первого ответа до решения: **7.6 часов**
- Медианное время: **6.3 часа**
- Максимальное время: **23.5 часа**

## 📝 Методология анализа

### 1. Профилирование данных
- Проверка качества данных (пропуски, дубликаты, аномалии)
- Анализ типов данных и их соответствия бизнес-логике

### 2. Исследовательский анализ (EDA)
- Распределение тикетов по статусам, приоритетам, типам
- Анализ временных метрик
- Выявление корреляций и паттернов

### 3. Формирование требований
- Бизнес-требования на основе выявленных проблем
- Пользовательские истории (User Stories)
- Нефункциональные требования

### 4. Моделирование процессов
- BPMN-схема процесса As-Is (текущее состояние)
- BPMN-схема процесса To-Be (целевое состояние)

## 💡 Предложенные решения

### Решение 1: Автоматические напоминания клиентам
- Email-напоминание через 24 часа в статусе "Pending"
- Повторное напоминание через 48 часов
- Автозакрытие через 7 дней без ответа

### Решение 2: Логирование создания тикетов
- Добавление поля `Created At` (TIMESTAMP)
- Расчёт полного времени обработки для SLA
- Дашборд с метриками производительности

### Решение 3: SLA и эскалация
- Мониторинг времени в каждом статусе
- Эскалация для High через 4 часа
- Эскалация для Critical через 2 часа
- Визуализация тикетов, приближающихся к нарушению SLA

## 📊 Метрики успеха (KPI)
1. Сокращение backlog с 67% до 30% за 3 месяца
2. 80% тикетов в "Pending" закрываются автоматически через 7 дней
3. 90% тикетов High решаются за 4 часа
4. Среднее время решения сокращается с 7.6 до 4 часов
